# AGENTS.md

## 1. このリポジトリの目的

このリポジトリは、SIGNATE「AI Engineering Challenge ～煩雑な社内ドライブをハックせよ～」向けのRAGシステムを開発するためのものです。

提供された共有ドライブ内の未加工ファイルと質問文を入力として、関連資料の検索、根拠抽出、必要な計算・集計、最終回答生成までを自動実行できる汎用的なシステムを構築します。

未知の案件フォルダ、未知のファイル、未知の質問が追加された場合でも、同じ処理方針で対応できることを最優先とします。

---

## 2. 最優先ルール

以下は、すべての実装より優先されるルールです。

- 特定の質問、案件、会社、ファイルにだけ対応する処理を実装しない。
- 正解、回答候補、ページ番号、セル番地、担当者名、金額などを手入力しない。
- `record_id`、質問文、会社名、ファイル名を使って固定回答を返さない。
- 人が事前に整理した回答用データベースや一覧表を作らない。
- 共有ドライブ内の原本ファイルを直接編集しない。
- 回答根拠として外部Web情報を使用しない。
- 質問への回答は、共有ドライブ内の資料から取得した根拠に基づいて生成する。
- 新しいデータに対しても適用できる、ファイル形式や文書構造に基づく汎用処理を実装する。
- 実行結果を再現できるように、設定、モデル名、乱数seed、ログを記録する。

案件名やファイル名による分岐ではなく、拡張子、MIMEタイプ、文書構造、質問意図などの一般化可能な情報を使って処理を選択してください。

## 3. データの取り扱い

### 入力データ

共有ドライブの原本は読み取り専用として扱います。

想定する入力ルートは環境変数で指定します。

```text
SOURCE_ROOT=/path/to/share/共有ドライブ
```

コード内に、開発者固有の絶対パスを書かないでください。

### 生成データ

自動生成したデータは、原本と分離して以下へ保存します。

```text
artifacts/
├── extracted/
├── normalized/
├── indexes/
├── cache/
└── runs/
```

ログは以下へ保存します。

```text
logs/
```

以下はGit管理しません。

- 共有ドライブの原本
- 抽出済みテキスト
- OCR結果
- 検索インデックス
- キャッシュ
- 実行ログ
- APIキー
- `.env`
- `predictions.csv`
- 提出用zipファイル

ただし、空ディレクトリ維持用の`.gitkeep`は管理対象とします。

## 4. 想定アーキテクチャ

処理は以下の責務に分離します。

```text
ファイル探索
  ↓
ファイル形式判定
  ↓
テキスト・表・画像・メタデータ抽出
  ↓
共通形式への正規化
  ↓
チャンク分割
  ↓
検索インデックス構築
  ↓
質問解析
  ↓
検索・再ランキング
  ↓
根拠整理
  ↓
回答生成
  ↓
提出形式への変換
```

推奨するパッケージ構成は以下です。

```text
src/signate_drive_rag/
├── cli.py
├── config.py
├── domain/
├── ingestion/
├── normalization/
├── indexing/
├── retrieval/
├── generation/
├── evaluation/
└── submission/
```

各モジュールは、できるだけ単一の責務を持たせてください。

外部ライブラリ固有のオブジェクトをシステム全体へ直接渡さず、必要に応じて`domain/`の共通データモデルへ変換してください。

## 5. 開発環境

- Python: 3.11
- パッケージ管理: `uv`
- Lint・Format: `ruff`
- 型チェック: `mypy`
- テスト: `pytest`
- 設定管理: `pyproject.toml`
- 秘密情報管理: `.env`
- バージョン管理: Git

依存関係を追加するときは、原則として`uv add`を使用します。

```bash
uv add <package>
uv add --dev <package>
```

`pip install`だけを実行して、`pyproject.toml`や`uv.lock`へ反映されない状態を作らないでください。

## 6. 実装ルール

### Python

- Python 3.11で動作するコードを書く。
- すべての公開関数・メソッドに型ヒントを付ける。
- `typing.List`や`typing.Dict`ではなく、`list`、`dict`、`tuple`などの組み込みジェネリクスを優先する。
- パス操作には`pathlib.Path`を使用する。
- データ構造には、必要に応じて`dataclass`またはPydanticモデルを使用する。
- 例外を握りつぶさない。
- `except Exception`を使用する場合は、理由を明確にし、ログを残す。
- グローバルな可変状態を避ける。
- I/O処理と純粋な変換処理を分離する。
- 外部サービスへの依存はインターフェースやクラスで分離し、テスト時に差し替えられるようにする。

### コメント・docstring

- docstringとコメントは日本語で記載する。
- コードそのものから分かる処理内容を、コメントで繰り返さない。
- コメントには、実装方法よりも「なぜこの判断をしたか」を残す。
- 制約、例外処理、精度上の判断、再現性に関わる理由を優先して記録する。

### 命名

- 変数名、関数名、クラス名、ファイル名は英語とする。
- 意味の曖昧な略称を避ける。
- `data`、`info`、`result`だけのような広すぎる名前を避ける。
- 真偽値には`is_`、`has_`、`should_`などを使用する。

## 7. ファイル解析ルール

ファイル形式ごとの処理は、拡張子またはMIMEタイプを基準に選択します。

想定する形式は以下です。

- PDF
- 画像PDF
- Word
- Excel
- CSV
- PowerPoint
- Markdown
- JSON
- Python
- Jupyter Notebook
- 画像

新しい形式を追加するときは、特定ファイル専用の分岐ではなく、パーサーを登録できる構造にしてください。

例:

```text
ParserRegistry
├── PdfParser
├── WordParser
├── ExcelParser
├── PowerPointParser
└── TextParser
```

抽出結果には、可能な範囲で以下のメタデータを保持します。

- 原本ファイルパス
- ファイル名
- 拡張子
- MIMEタイプ
- ページ番号
- シート名
- スライド番号
- セル範囲
- 見出し階層
- 更新日時
- 抽出方式
- OCR使用有無

根拠を回答へ結び付けられるように、正規化やチャンク分割後も原本位置を追跡できる設計にしてください。

## 8. テストルール

新しい機能を追加するときは、対応するテストを追加します。

### Unit test

以下を中心に確認します。

- 入力と出力
- 境界値
- 空データ
- 不正なデータ
- 日本語ファイル名
- 日本語ディレクトリ名
- 存在しないパス
- 対応外ファイル形式

### Integration test

以下を確認します。

- ファイル探索から抽出まで
- 抽出から正規化まで
- インデックス構築から検索まで
- 質問から回答生成まで
- 提出ファイル生成

テストでは、共有ドライブの実データに直接依存せず、`tests/fixtures/`の小さなサンプルデータを使用してください。

テスト名から、何を保証しているか分かるようにします。

```python
def test_discover_files_includes_nested_japanese_paths() -> None:
    ...
```

## 11. ログと再現性

各実行では、可能な範囲で以下を記録します。

- 実行日時
- Git commit hash
- 使用した設定
- Pythonバージョン
- 主要ライブラリのバージョン
- 使用モデル名
- 埋め込みモデル名
- seed
- 入力ファイル数
- 処理成功数
- 処理失敗数
- エラー内容
- 出力先

LLMやOCRなど出力が揺れる処理では、設定可能な場合はseedを固定します。

失敗したファイルがあっても、全体を停止すべきか処理を継続すべきかを明確に判断し、処理結果へ記録してください。

## 12. Git運用

作業は最新の`main`からブランチを作成して行います。

```bash
git switch main
git pull origin main
git switch -c <type>/<task-name>
```

ブランチ名の例:

```text
feat/add-file-discovery
fix/handle-empty-pdf
docs/update-agents-guidelines
test/add-excel-parser-tests
chore/update-dependencies
```

コミットメッセージは、変更内容が分かる英語で記載します。

```text
feat: add recursive file discovery
fix: preserve Excel sheet metadata
docs: define agent development rules
test: cover unsupported file types
chore: update development dependencies
```

コミットは、関連する変更単位で分けてください。

コードコメントは「Why」、コードは「How」、テストは「What」、コミットメッセージは変更の意図が分かるように記載します。

## 13. Definition of Done

以下を満たした場合に、作業完了とします。

- 要求された機能が実装されている。
- 特定の質問、会社、ファイルに依存するハードコードがない。
- 新しい案件やファイルにも適用できる設計になっている。
- 型ヒントが追加されている。
- 必要なdocstringが追加されている。
- 対応するテストが追加されている。
- `ruff`が成功する。
- `mypy`が成功する。
- `pytest`が成功する。
- 秘密情報や提供データがGit管理に含まれていない。
- READMEまたは関連ドキュメントが必要に応じて更新されている。
- 実行方法と設定方法が再現可能である。

## 14. エージェントへの指示

実装を開始する前に、以下を行ってください。

1. 関連する既存コード、テスト、設定を確認する。
2. 変更範囲を最小限に整理する。
3. 特定データへの過剰適合やハードコードがないか確認する。
4. 実装する。
5. テストを追加または更新する。
6. 品質チェックを実行する。
7. 変更内容、確認結果、残課題を報告する。

不明点がある場合でも、既存コードと本ファイルから合理的に判断できる範囲は、自律的に進めてください。

大規模な設計変更、依存関係の大量追加、データ形式の変更、既存インターフェースの破壊的変更を行う前には、その必要性と影響範囲を明示してください。
