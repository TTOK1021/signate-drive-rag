# SIGNATE Drive RAG

SIGNATE「煩雑な社内ドライブをハックせよ」向けのRAGシステム開発リポジトリです。

実務の共有ドライブに近い多様な資料群を対象に、質問文から必要な根拠資料を検索・読解し、根拠に基づく最終回答を生成するAgentic RAGパイプラインの構築を目的とします。

## コンペ概要

参加者は、架空のデータ分析コンサルティング会社「データアステル社」の社内AI担当として、共有ドライブに蓄積された案件資料から必要な情報を探し出し、社内から寄せられる質問に正確に回答するRAGシステムを構築します。

対象資料には、テキストだけでなく、PDF、画像PDF、Excel、CSV、JSON、Notebook、Pythonコード、PowerPoint、Word、画像、グラフ、表などが含まれます。そのため、単純なテキスト検索ではなく、文書構造、表示上の情報、社内用語、案件間の関係、複数資料にまたがる照合・集計を扱う必要があります。

重要なのは、特定の質問や資料にだけ最適化した実装ではなく、未知の案件フォルダ、未知の資料、未知の質問が追加された場合でも同じ処理方針で動く汎用的なRAGシステムを作ることです。

## 主要要件

- 共有ドライブ内の未加工データと質問文のみを入力として回答を生成する
- 複数形式のファイルを、形式に応じた汎用処理で解析する
- テキスト、表、OCR結果、セル値、書式情報、画像情報、メタデータ、グラフ情報などを機械的に抽出する
- ベクトルDB、全文検索、SQL、DataFrame、ナレッジグラフなどを検索インデックスや中間表現として利用する
- 質問に応じて関連資料を検索し、抽出した根拠情報をLLMや回答生成処理に渡す
- 社内用語集を検索対象に含め、必要に応じて社内用語・略称を通常表現へ展開する
- 計算、集計、比較、差分確認、条件抽出、ランキング作成などをプログラムで実行する
- コードで再現可能な処理として構築し、再実行時にも同等または近い回答を再現する

## 回答時の注意

- 回答形式、単位、小数桁、丸め方、主略称の指定、抽出対象の表記などは質問文の条件に従う
- 設問で明示されている場合を除き、社内用語・略称ではなく通常の表現で回答する
- タスクID、アクションID、マイルストーンID、列名、パラメータ名などの識別子は資料上の表記どおりに回答する
- 条件に該当する情報や対象が存在しない場合は、該当するものがない旨を回答する
- 1回答あたりの最大文字数は1000トークン
- 応答時間の制限は3時間

## 評価方法

各質問への回答は、LLMにより以下の4分類で評価されます。

| 分類 | 得点 | 評価基準 |
| --- | ---: | --- |
| Perfect | 1 | 質問に正確に答え、虚偽の内容が含まれていない回答 |
| Acceptable | 0.5 | 有用な答えを提供しているが、有用性を損なわない程度の軽微な誤りがある回答 |
| Missing | 0 | 「わかりません」「見つけられませんでした」など、具体的な答えがない回答 |
| Incorrect | -1 | 間違った回答、または質問と関連性のない回答 |

最終スコアは、各質問の得点合計を質問数で割った平均値です。スコア範囲は -1 から 1 です。

評価では、temperatureは0、seedも固定されます。ただし、微妙な回答内容ではLLM判断にわずかな揺らぎが発生する可能性があります。

## 禁止事項

以下のような、人手による事前加工や特定問題への個別最適化は禁止されています。

- 提供データを人手で整理し、回答しやすい独自DBや一覧表を作って読み込ませる
- 質問ごとの正解候補、参照ファイル、重要ページ、重要セル、抽出条件などを人手でまとめたファイルを使う
- 特定質問への回答、特定案件の金額・担当者名・ページ番号・セル番地などをコード、プロンプト、設定ファイル、辞書、ルールベース処理に書く
- 質問文やrecord_idを見て固定回答を返す
- 案件名やファイル名を見て専用分岐を作る
- 質問ごとに人がPDF、Excel、PowerPoint、NotebookLMなどを開いて回答を作成する
- GUIツールに質問を1問ずつ入力し、その出力を提出回答として使う
- 提供資料を検索・読解せず、外部知識、推測、LLMの事前知識だけで回答する
- 外部Web検索で取得した提供資料外の情報を回答根拠として使用する
- 評価プロンプトをハックするような指示を投稿に含める

## データとツール利用

回答根拠として利用できる情報は、提供された共有ドライブ内のフォルダ・ファイル群に含まれる情報に限られます。

LLMモデルの学習、ファインチューニング、評価、企業実装などに外部データを使う場合は、商業利用が可能なデータに限り利用できます。外部データを利用する場合は、データソースやライセンス情報などを最終提出物に明記する必要があります。

使用可能なツールは、商業利用が可能でオープンなものに限定されます。Python、R、TensorFlow、外部ライブラリ、OCRエンジン、LLM API、埋め込みモデル、検索エンジン、表抽出ツール、マルチモーダルモデル、エージェントフレームワークなどは利用可能です。

## 質問データ

質問データは以下に配置されています。

```text
share/質問回答/
├─ questions_valid.csv
└─ questions_test.csv
```

検証用・提出用の回答生成では、これらの質問CSVと `share/共有ドライブ/` 配下の未加工資料を入力として扱います。

## 提出形式

提出ファイルは、必ず `predictions.csv` という名前のCSVファイルにします。

フォーマットはヘッダーなしで、以下のように問題番号と回答テキストを並べます。

| index | answer |
| ---: | --- |
| 0 | 回答文 |
| ... | ... |

`predictions.csv` をzip形式に圧縮して提出します。zipファイル名は任意ですが、英数字、アンダースコア、ハイフンのみの使用が想定されています。

## ローカル評価環境

`evaluation/` には、回答CSVのフォーマット検証とLLM評価を行うためのスクリプトが含まれています。

```text
evaluation/
├─ data/
│  └─ valid_txt.csv
├─ src/
│  ├─ dbmanager.py
│  ├─ evaluator.py
│  ├─ settings.py
│  └─ validator.py
├─ submit/
│  └─ predictions.csv
├─ crag.py
├─ docker-compose.yml
├─ Dockerfile
└─ readme.md
```

主な処理は以下です。

- `crag.py`: 評価実行の入口
- `src/settings.py`: 評価対象サンプル、拡張子、最大トークン数などの評価フォーマットを設定
- `src/validator.py`: CSV拡張子、サンプル数、欠損、型、トークン数を検証
- `src/evaluator.py`: OpenAI APIを使い、回答を `Perfect` / `Acceptable` / `Missing` / `Incorrect` に分類
- `src/dbmanager.py`: 正解データの読み込みと `scoring.csv` の保存

## 評価環境の起動

Dockerを使う場合は、`evaluation/` 配下で以下を実行します。

```bash
docker compose up -d
docker exec -it rag_env bash
```

コンテナ内でOpenAI APIキーを設定します。

```bash
export OPENAI_API_KEY={YOUR_API_KEY}
```

Dockerを使わない場合は、概ね以下のライブラリが必要です。

```text
numpy==1.26.4
pandas==2.2.2
openai==1.57.1
tiktoken==0.7.0
torch==2.3.1+cpu
sentence-transformers==3.0.1
```

## 評価の実行

`evaluation/` 配下で以下を実行します。

```bash
python crag.py
```

デフォルト設定では、以下を入力として使います。

- 正解データ: `evaluation/data/valid_txt.csv`
- 予測回答: `evaluation/submit/predictions.csv`
- 最大トークン数: 1000
- 評価モデル: `gpt-5.2-2025-12-11`
- 評価結果出力先: `evaluation/result/scoring.csv`

評価結果の `scoring.csv` は、問題番号、評価結果、回答トークン数を含みます。

## 実装方針メモ

このリポジトリでは、以下のようなパイプラインを目指します。

1. 共有ドライブ配下のファイルを再帰的に走査する
2. ファイル形式ごとにテキスト、表、画像、グラフ、メタデータを抽出する
3. 社内用語集を含めた検索インデックスを作成する
4. 質問文から回答条件、対象資料、必要な操作を解析する
5. 関連資料を検索し、必要に応じて計算・集計・比較を行う
6. 根拠情報をもとに、指定形式に合わせて最終回答を生成する
7. 回答、根拠、使用モデル、ログを保存して再現性を担保する

LLMやOCRの揺らぎがある場合は、可能な範囲でseed固定、ログ保存、使用モデル名の記録を行います。

## Git管理方針

`.gitignore` では、以下を主に除外しています。

- `.env` などの秘密情報
- `.venv` やPythonキャッシュ
- コンペ提供データの `share/`
- 評価用データの `evaluation/data/`
- `artifacts/` と `logs/` の生成物
- 提出CSVとzipファイル

`artifacts/.gitkeep` と `logs/.gitkeep` は、空ディレクトリを維持するために管理対象にしています。
